# 孕期体重助手 · 产品架构设计

> 基于 `PRD.md` v0.1  
> 产品形态：Web PWA / 本地存储 / 个人自用 / 无 AI / 无服务器

---

## 1. 产品架构

### 1.1 产品目标

孕期体重助手是一个面向准妈妈个人使用的轻量级 PWA 工具，核心目标是帮助用户用最低操作成本完成每日体重打卡，并结合孕周、孕前 BMI 与中国行业标准区间，提供低焦虑、可视化的体重趋势参考。

产品不承担医疗诊断职责，不提供处方建议，不依赖服务器，不设计账号体系。v1 的关键价值是“打开即记录、趋势可看、数据可备份”。

### 1.2 产品定位校准

产品设计理念：

1. 极快：每天打开应用后，10 秒内完成记录并关闭。
2. 极简：主页只服务“记录体重”这一件事，其他能力收进趋势页和设置页。
3. 无广告：不接广告位，不做商业推荐，不出现打扰型横幅。
4. 无社区：不做动态、评论、关注、分享广场。
5. 无打卡排名：不做连续打卡榜、体重控制榜、同类对比榜。
6. 无焦虑营销：不使用恐吓、攀比、催促、倒计时式焦虑文案。

核心使用目标：

1. 打开。
2. 记录。
3. 关闭。

主页体验必须优先满足“10 秒完成一次体重打卡”。宝宝发育、趋势查看、数据备份是辅助能力，不应抢占主页的主要注意力。

### 1.3 核心用户路径

1. 首次打开应用。
2. 填写预产期、孕前身高、孕前体重。
3. 系统计算当前孕周、孕天、剩余天数、孕前 BMI 与 BMI 分类。
4. 进入主页，输入今日体重并保存。
5. 保存后展示今日体重相对上周均值的变化。
6. 关闭应用。
7. 需要时再进入趋势页查看实际增重曲线与标准区间对比。
8. 需要备份时进入设置页导出或导入 CSV。

### 1.4 产品模块划分

| 模块 | 职责 | 优先级 |
| --- | --- | --- |
| 首次引导模块 | 收集预产期、孕前身高、孕前体重 | P0 |
| 孕周计算模块 | 计算孕周、孕天、剩余天数、280 天进度 | P0 |
| 体重打卡模块 | 输入、微调、备注、保存当日记录 | P0 |
| 反馈模块 | 保存后展示今日 vs 上周均值 | P0 |
| 趋势图模块 | SVG 绘制实际曲线与标准区间色带 | P0 |
| 标准区间模块 | 基于 WS/T 809-2022 和 BMI 分类计算推荐区间 | P0 |
| 宝宝发育模块 | 按孕周展示果蔬比喻和科普文案 | P1 |
| 数据管理模块 | CSV 导出、粘贴导入、文件上传导入 | P1 |
| PWA 模块 | manifest、离线缓存、添加到主屏幕 | P1 |

### 1.5 技术架构

采用纯前端单页应用架构。

| 层级 | 说明 |
| --- | --- |
| UI 层 | React 页面、组件、底部 Tab、表单、图表、列表 |
| 状态层 | React state / context，管理 profile、records、当前 Tab、导入状态 |
| 业务服务层 | 孕周计算、BMI 分类、标准区间计算、周均值计算、CSV 解析导出 |
| 持久化层 | localStorage，保存个人信息和体重记录 |
| 静态数据层 | WS/T 809-2022 参数表、宝宝发育文案 |
| PWA 层 | manifest、service worker、静态资源缓存 |

### 1.6 架构原则

1. 本地优先：所有个人数据默认只保存在浏览器本地。
2. 极简操作：主页必须承担最核心的打卡任务，避免复杂导航。
3. 医疗边界清晰：只做标准区间参考，不做诊断和处方。
4. 可恢复：必须提供 CSV 导出和导入，降低 localStorage 丢失风险。
5. 离线可用：核心打卡、查看历史、查看趋势应在无网络时可用。
6. 移动优先：手机浏览器打开即用，底部 Tab 作为主导航。
7. 零增长机制：不引入广告、社区、排名、连续打卡压力或运营弹窗。
8. 主页优先级约束：保存体重按钮、数字输入和微调按钮永远是主页最高优先级。

## 2. 页面结构

### 2.0 UI 设计原则

整体视觉方向：

1. Natural Tones。
2. 森林系。
3. 不使用粉色。
4. 不使用医疗蓝。
5. 不使用广告感强、促销感强、高刺激的配色。

基础 UI 规范：

| 项 | 规范 |
| --- | --- |
| 字体 | `Noto Serif SC` |
| 按钮圆角 | 16px |
| 主色方向 | 鼠尾草绿、苔藓绿、橄榄绿、暖白、浅木色 |
| 禁用色方向 | 粉色、荧光色、医疗蓝、高饱和警示色 |
| 视觉气质 | 安静、自然、克制、柔和 |
| 动效 | 轻量，避免强提醒和持续闪烁 |

页面设计约束：

1. 首页首屏必须优先展示体重输入和保存动作。
2. 不放广告位、运营横幅、社区入口、排名入口。
3. 不使用“快来坚持”“超过多少人”“落后了”等压力文案。
4. 趋势页只表达“参考区间”和“趋势”，不制造合格或失败感。
5. 设置页的数据风险提示要清楚，但语气保持平静。

### 2.1 全局结构

应用采用单页三 Tab 架构。

| 区域 | 内容 |
| --- | --- |
| 顶部状态区 | 当前孕周、孕天、剩余天数、280 天进度 |
| 主内容区 | 根据当前 Tab 展示主页、趋势或设置 |
| 底部导航 | 主页 / 趋势 / 设置 |

首次使用时，如果缺少 `pregnancy_profile`，进入首次引导流程。完成后进入主页。

### 2.2 Tab 1：主页 / 打卡

页面目标：打开后立即完成今日体重记录。

结构：

1. 孕周进度条
   - 当前孕周。
   - 当前孕天。
   - 距离预产期剩余天数。
   - 280 天进度条。

2. 今日打卡仪表盘
   - 当前日期。
   - 体重输入框，单位 kg，精度 0.1。
   - 快速微调按钮：-0.5、-0.1、+0.1、+0.5。
   - 备注快捷标签：晨起空腹、晚餐后、浮肿日。
   - 保存按钮。
   - 如果当天已有记录，输入框默认回填当天最新记录。

3. 保存反馈
   - 今日体重。
   - 上周均值。
   - 差值。
   - 温和提示文案，避免制造焦虑。

4. 宝宝本周发育卡片
   - 当前孕周。
   - 果蔬大小比喻。
   - 一条发育科普。

### 2.3 Tab 2：趋势 / 图表

页面目标：让用户看到自己的增重趋势与标准区间关系。

结构：

1. 增重趋势图
   - 使用 SVG 绘制。
   - x 轴：孕周。
   - y 轴：相对孕前体重的增重 kg。
   - 标准区间色带。
   - 实际周均值曲线。
   - 记录节点。
   - 点击节点显示当周均值和状态。

2. 周详情浮层或详情区
   - 孕周。
   - 当周均值。
   - 相对孕前增重。
   - 标准区间。
   - 状态：正常 / 偏低 / 偏高。

3. 历史记录列表
   - 按日期倒序。
   - 日期。
   - 孕周。
   - 体重。
   - 备注。
   - 同日多次记录只展示最新值。

### 2.4 Tab 3：设置 / 数据

页面目标：维护个人基础信息和本地数据备份。

结构：

1. 个人信息
   - 预产期。
   - 孕前身高。
   - 孕前体重。
   - 自动计算孕前 BMI。
   - 自动显示 BMI 分类。

2. 数据管理
   - 导出 CSV。
   - 粘贴 CSV 导入。
   - 文件上传导入。
   - 导入前预览。
   - 导入冲突处理策略说明。

3. 本地数据说明
   - 数据仅保存在当前浏览器。
   - 清理浏览器数据、更换设备或隐私模式可能导致数据丢失。

## 3. 数据库设计

v1 不引入服务器数据库。数据库设计对应浏览器本地存储，主要使用 `localStorage`。

### 3.1 localStorage 键名

| key | 类型 | 说明 |
| --- | --- | --- |
| `pregnancy_profile` | object | 用户个人孕期基础信息 |
| `pregnancy_records` | array | 体重打卡记录 |
| `pregnancy_app_meta` | object | 可选，应用版本、首次使用时间、最近备份时间 |

### 3.2 个人信息：pregnancy_profile

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `dueDate` | string | 是 | 预产期，格式 `YYYY-MM-DD` |
| `heightCm` | number | 是 | 孕前身高，单位 cm |
| `preWeightKg` | number | 是 | 孕前体重，单位 kg |
| `preBMI` | number | 是 | 孕前 BMI，保留 1 位小数 |
| `bmiCategory` | string | 是 | BMI 分类：`underweight` / `normal` / `overweight` / `obese` |
| `updatedAt` | number | 建议 | 最近更新时间，毫秒时间戳 |

### 3.3 体重记录：pregnancy_records

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `date` | string | 是 | 打卡日期，格式 `YYYY-MM-DD` |
| `weightKg` | number | 是 | 体重，单位 kg，精度 0.1 |
| `note` | string | 否 | 备注标签或自定义备注 |
| `createdAt` | number | 是 | 创建或覆盖保存时间，毫秒时间戳 |

### 3.4 派生数据

以下数据不需要持久化，应在运行时由 profile 和 records 计算得到：

| 数据 | 计算来源 |
| --- | --- |
| 当前孕周、孕天 | `dueDate` 与当前日期 |
| 剩余天数 | `dueDate` 与当前日期 |
| 280 天进度 | 当前孕天 / 280 |
| 孕前 BMI | `heightCm` 和 `preWeightKg` |
| BMI 分类 | `preBMI` |
| 增重值 | 当前体重 - 孕前体重 |
| 周均值 | 同孕周记录平均值 |
| 上周均值 | 当前孕周 - 1 的记录平均值 |
| 趋势状态 | 实际增重值与标准区间对比 |

### 3.5 同日多次记录策略

PRD 要求“同日多次记录取最新值”。推荐策略：

1. 保存时以 `date` 为唯一键。
2. 如果当天不存在记录，则新增。
3. 如果当天已存在记录，则覆盖 `weightKg`、`note` 和 `createdAt`。
4. 历史列表和图表均基于覆盖后的最新值。

### 3.6 CSV 数据格式

导出建议字段：

| 字段 | 说明 |
| --- | --- |
| `date` | 日期 |
| `weightKg` | 体重 kg |
| `note` | 备注 |
| `createdAt` | 时间戳 |

CSV 导出必须添加 UTF-8 BOM，避免中文备注在 Excel 中乱码。

## 4. API 设计

v1 无服务端 API。这里的 API 指前端内部业务服务接口和浏览器能力接口。

### 4.1 孕周计算服务

| 方法 | 入参 | 返回 | 说明 |
| --- | --- | --- | --- |
| `calculatePregnancyProgress` | `dueDate`, `today` | 孕周、孕天、剩余天数、进度 | 基于预产期倒推 280 天 |
| `getGestationalWeekByDate` | `dueDate`, `date` | 孕周 | 用于历史记录和图表聚合 |

业务规则：

1. 怀孕起始日 = 预产期 - 280 天。
2. 当前孕天 = 当前日期 - 怀孕起始日。
3. 当前孕周 = floor(当前孕天 / 7) + 1。
4. 边界值应限制在 1 到 40 周附近，避免图表异常。

### 4.2 BMI 服务

| 方法 | 入参 | 返回 | 说明 |
| --- | --- | --- | --- |
| `calculateBMI` | 身高 cm、体重 kg | BMI | 保留 1 位小数 |
| `getBMICategory` | BMI | BMI 分类 | 用于选择标准区间 |

建议分类：

| 分类 | 条件 |
| --- | --- |
| `underweight` | BMI < 18.5 |
| `normal` | 18.5 <= BMI < 24 |
| `overweight` | 24 <= BMI < 28 |
| `obese` | BMI >= 28 |

### 4.3 体重记录服务

| 方法 | 入参 | 返回 | 说明 |
| --- | --- | --- | --- |
| `getRecords` | 无 | 记录数组 | 从 localStorage 读取 |
| `saveRecord` | date、weightKg、note | 记录数组 | 新增或覆盖当天记录 |
| `getTodayRecord` | date | 记录或空 | 查询当天最新记录 |
| `getRecordsByWeek` | records、profile | 周聚合数据 | 图表使用 |
| `getPreviousWeekAverage` | records、当前孕周 | number 或空 | 保存反馈使用 |

### 4.4 标准区间服务

| 方法 | 入参 | 返回 | 说明 |
| --- | --- | --- | --- |
| `getStandardRange` | BMI 分类、孕周 | 最小增重、最大增重 | 根据 WS/T 809-2022 参数表 |
| `getWeightStatus` | 实际增重、标准区间 | 正常 / 偏低 / 偏高 | 图表节点状态 |
| `buildStandardBand` | BMI 分类、周范围 | SVG 色带点位 | 趋势图使用 |

说明：标准数据应作为静态 JSON 参数表维护，避免散落在组件里。

### 4.5 CSV 服务

| 方法 | 入参 | 返回 | 说明 |
| --- | --- | --- | --- |
| `exportRecordsToCSV` | records | CSV 字符串 | 添加 UTF-8 BOM |
| `parseCSVText` | CSV 文本 | 记录数组 | 粘贴导入 |
| `readCSVFile` | File | CSV 文本 | 文件上传导入 |
| `mergeImportedRecords` | 本地记录、导入记录 | 合并后记录 | 同日期保留较新的 `createdAt` |

导入校验：

1. 日期必须合法。
2. 体重必须为正数。
3. 体重精度最多 1 位小数。
4. 空备注允许。
5. 非法行应给出提示，不应导致整体导入失败。

### 4.6 本地存储服务

| 方法 | 入参 | 返回 | 说明 |
| --- | --- | --- | --- |
| `loadProfile` | 无 | profile 或空 | 读取个人信息 |
| `saveProfile` | profile | void | 写入个人信息 |
| `loadRecords` | 无 | records | 读取记录 |
| `saveRecords` | records | void | 写入记录 |
| `clearAllData` | 无 | void | 可选功能，v1 谨慎提供 |

## 5. React 组件设计

### 5.1 组件分层

| 层级 | 组件类型 | 说明 |
| --- | --- | --- |
| App 层 | `App` | 加载数据、判断首次引导、维护当前 Tab |
| 页面层 | `HomePage` / `TrendPage` / `SettingsPage` / `OnboardingPage` | 承接页面布局和业务组合 |
| 业务组件层 | 孕周进度、体重打卡、趋势图、历史列表、导入导出 | 封装具体业务 |
| 基础组件层 | Button、Input、Tabs、Card、Toast、Modal | 可复用 UI |
| 服务层 | storage、pregnancy、bmi、records、csv、standardRange | 与 React 解耦 |

### 5.2 App 组件

职责：

1. 初始化读取 `pregnancy_profile` 和 `pregnancy_records`。
2. 如果缺少 profile，展示首次引导。
3. 维护当前底部 Tab。
4. 将 profile 和 records 下发给页面。
5. 处理全局保存、更新和刷新。

### 5.3 OnboardingPage

职责：

1. 收集预产期。
2. 收集孕前身高。
3. 收集孕前体重。
4. 计算 BMI 和 BMI 分类。
5. 保存 profile。
6. 完成后进入主页。

关键状态：

| 状态 | 说明 |
| --- | --- |
| `dueDate` | 预产期 |
| `heightCm` | 孕前身高 |
| `preWeightKg` | 孕前体重 |
| `errors` | 表单校验错误 |

### 5.4 HomePage

包含组件：

| 组件 | 职责 |
| --- | --- |
| `PregnancyProgressHeader` | 显示孕周、孕天、剩余天数、进度条 |
| `WeightCheckInPanel` | 体重输入、微调、备注、保存 |
| `SaveFeedback` | 今日 vs 上周均值 |
| `BabyDevelopmentCard` | 本周宝宝发育 |

### 5.5 WeightCheckInPanel

职责：

1. 默认填入今日已有记录或最近一次体重。
2. 支持数字键盘输入。
3. 支持 0.1 和 0.5 微调。
4. 支持备注标签选择。
5. 保存时执行输入校验。
6. 保存成功后通知父组件刷新 records。

校验规则：

1. 体重不能为空。
2. 体重必须为合理正数。
3. 体重最多 1 位小数。
4. 日期默认为当前本地日期。

### 5.6 TrendPage

包含组件：

| 组件 | 职责 |
| --- | --- |
| `WeightTrendChart` | SVG 绘制标准区间和实际趋势 |
| `WeekDetailPanel` | 展示选中节点详情 |
| `RecordHistoryList` | 按日期倒序展示记录 |

### 5.7 WeightTrendChart

职责：

1. 将 records 聚合为孕周均值。
2. 将标准区间转换为 SVG 色带。
3. 绘制实际增重曲线。
4. 绘制节点。
5. 响应节点点击或触摸。
6. 为空数据状态提供友好展示。

注意：PRD 要求不使用第三方图表库，因此图表坐标转换、路径生成和触摸区域需要自行封装。

### 5.8 SettingsPage

包含组件：

| 组件 | 职责 |
| --- | --- |
| `ProfileSettingsForm` | 编辑预产期、孕前身高、孕前体重 |
| `DataExportPanel` | 导出 CSV |
| `DataImportPanel` | 粘贴导入、文件上传导入 |
| `LocalDataNotice` | 本地存储风险提示 |

### 5.9 状态管理建议

v1 不需要引入复杂状态管理库。建议使用：

1. React 内置 state 管理当前页面和表单状态。
2. 自定义 hook 管理 profile 与 records。
3. 服务函数处理业务计算。
4. localStorage 作为唯一持久化来源。

## 6. 开发顺序

### 6.1 第一阶段：可用原型

目标：用户可以开始真实打卡。

1. 初始化 Vite + React + TypeScript 项目。
2. 配置 Tailwind CSS 和 Natural Tones 色彩。
3. 搭建 App 外壳、三 Tab 布局和移动端基础样式。
4. 完成首次引导表单。
5. 完成 profile 本地存储。
6. 完成孕周计算和顶部进度条。
7. 完成体重输入、微调、备注标签和保存。
8. 完成 records 本地存储。
9. 完成同日覆盖策略。
10. 完成今日 vs 上周均值反馈。

### 6.2 第二阶段：趋势闭环

目标：用户可以理解自己的体重趋势。

1. 整理 WS/T 809-2022 标准参数表。
2. 完成 BMI 分类与标准区间计算。
3. 完成 records 按孕周聚合。
4. 完成 SVG 坐标系统。

## 7. 风险点

### 7.1 医疗合规风险

风险：体重区间和状态提示可能被用户理解为医学诊断。

应对：

1. 文案明确为“参考”而非“诊断”。
2. 避免使用“异常”“危险”等强刺激词。
3. 偏高或偏低时提示“如持续明显偏离，建议咨询产检医生”。
4. v1 不提供 AI 诊断、处方或医学建议。

### 7.2 标准数据准确性风险

风险：WS/T 809-2022 参数整理错误会影响趋势判断。

应对：

1. 将标准参数集中维护为静态表。
2. 数据录入后人工复核。
3. 保留数据来源说明。
4. 对标准区间计算函数增加测试。

### 7.3 本地数据丢失风险

风险：localStorage 会因清理浏览器数据、更换设备、隐私模式或浏览器策略而丢失。

应对：

1. 设置页明确提示“数据只保存在当前浏览器”。
2. 提供 CSV 导出。
3. 提供 CSV 导入。
4. 可在用户连续使用一段时间后提示备份。

### 7.4 日期与孕周边界风险

风险：时区、日期格式、预产期边界可能导致孕周计算偏差。

应对：

1. 所有业务日期使用本地日期 `YYYY-MM-DD`。
2. 避免直接用包含时分秒的字符串参与孕周计算。
3. 统一封装日期计算服务。
4. 对预产期前后、跨年、当天边界增加测试。

### 7.5 SVG 图表复杂度风险

风险：不使用第三方图表库会增加坐标转换、缩放、点击区域和小屏适配成本。

应对：

1. 先实现静态图，再增加节点交互。
2. 将坐标转换函数与 React 组件拆开。
3. 小屏下减少轴标签密度。
4. 节点触摸区域大于视觉圆点。

### 7.6 导入数据污染风险

风险：用户粘贴错误 CSV、重复日期、异常体重或乱码内容，可能污染本地记录。

应对：

1. 导入前校验。
2. 导入前预览。
3. 非法行跳过并提示。
4. 同日期按较新的 `createdAt` 合并。
5. 导入前可自动创建当前数据快照。

### 7.7 PWA 缓存更新风险

风险：service worker 缓存旧版本资源，导致用户看到旧逻辑。

应对：

1. 使用明确的缓存版本号。
2. 新版本发布后提示刷新。
3. 避免缓存导入导出生成文件。
4. 发布后验证安装态和浏览器态。

### 7.8 情绪体验风险

风险：孕期体重数据容易引发焦虑，过度强调偏离会降低使用意愿。

应对：

1. 反馈文案保持温和。
2. 使用“趋势参考”而不是“是否合格”。
3. 首页优先展示完成打卡的正反馈。
4. 宝宝发育卡片提供情感连接，平衡数据压力。
